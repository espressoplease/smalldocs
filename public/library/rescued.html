Rescued files

When a Markdown file lives in a folder the operating system will clean up on its own schedule, SDocs takes a snapshot at the moment you opened it so the content survives.

What you see in the library

Entries marked rescued have a snapshot stored under your sdocs home directory (~/.sdocs/library/rescued/). The original location is still recorded so you can see where the file came from, but the snapshot is what gets opened when you click the entry.

Which folders trigger a rescue

Anything under a known throwaway location at the time of opening:

  • /tmp, /var/tmp (and their /private/... aliases on macOS)
  • ~/.Trash on macOS
  • /run, /dev/shm, ~/.cache on Linux

OS scratch directories like /var/folders/... on macOS or %TEMP% on Windows are skipped entirely. Test runners and build tools write there constantly; SDocs treats anything in those locations as not worth indexing.

How edits work

Clicking a rescued entry opens the snapshot. Edits write back to the snapshot, not the original location, because the original may already be gone. The snapshot is the source of truth from that point onward.

This is a snapshot, not a live mirror. If you edit the file in its original location after the first sdoc, the snapshot stays at the moment you opened it. To bring the snapshot up to date, open the file again.

Removing a rescued entry

Delete the snapshot under ~/.sdocs/library/rescued/ and run sdoc library rescan — the next scan drops entries whose snapshot is gone. Or just leave them; rescued snapshots are small.
